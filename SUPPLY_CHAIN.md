# Supply Chain Policy

This document records the supply-chain hardening posture of the bocpy
repository: what gets pinned, how it is verified in CI, and how to
regenerate the pinned artifacts. It is the source of truth for the
"why" behind `ci/constraints-*.txt`, `.github/dependabot.yml`, the
SHA-pinned GitHub Actions in `.github/workflows/`, and the
PEP 770 SBOMs shipped inside wheels.

## At a glance

| Surface                                  | Pinned to            | Verified by                        | Regenerated by                              |
|------------------------------------------|----------------------|------------------------------------|---------------------------------------------|
| Python deps (`[test]/[linting]/[docs]/[audit]`) | exact version + sha256 | `pip install --require-hashes` in every CI job + `pip-audit --strict` | `uv pip compile --universal --generate-hashes` (see below) |
| GitHub Actions (`uses:` lines)           | full 40-char SHA     | dependabot weekly PRs              | dependabot (manual: `git ls-remote <repo> <tag>`) |
| Downstream consumer template (`templates/c_abi_consumer`) | `bocpy~=MAJOR.MINOR` | `c-abi-consumer` job builds it     | `finalize-pr` skill                           |
| Released wheels                          | (PEP 770) CycloneDX 1.6 SBOM inside `*.dist-info/sboms/` | `verify_sboms` job (structural validator + `grype`) | `scripts/build_sbom.py` during cibuildwheel |

## Python dependency pinning

`bocpy` itself has **zero third-party runtime Python dependencies** —
it ships only stdlib usage and its own C extensions. All third-party
deps live in `[project.optional-dependencies]` in
[pyproject.toml](pyproject.toml) and are pinned per-extra in
[ci/](ci/):

* `ci/constraints-test.txt` — pytest, coverage, fixtures.
* `ci/constraints-linting.txt` — flake8 + plugins.
* `ci/constraints-docs.txt` — sphinx + plugins.
* `ci/constraints-audit.txt` — `pip-audit` (used only by the `audit`
  job in `.github/workflows/pr_gate.yml`).

Each file is produced by `uv pip compile` in **universal** mode so a
single lockfile per extra covers every supported interpreter
(Python 3.10 — 3.14, including free-threaded). PEP 508 environment
markers in the output handle minor-version drift (e.g.
`exceptiongroup ; python_full_version < '3.11'`).

### Why a two-step install

pip refuses to honour `--require-hashes` when an editable install is
part of the requirement set. Every CI job therefore uses the same
two-step pattern:

```yaml
- name: Install pinned <extra> dependencies
  run: pip install -r ci/constraints-<extra>.txt --require-hashes

- name: Install bocpy (editable, no deps)
  run: pip install -e . --no-deps
```

The dependency graph is locked and hash-verified by step 1; bocpy
itself is installed editable (so the in-tree C sources are compiled
against the matrix's interpreter) without touching the dependency
solver. There is no path by which an unpinned transitive dependency
can enter a CI job.

### Regenerating the constraints

The exact regen command is preserved in the first comment of every
generated file. Run all four together from the repository root:

```bash
for extra in test linting docs audit; do
    uv pip compile --universal --generate-hashes \
        --python-version 3.10 --extra "$extra" \
        pyproject.toml -o "ci/constraints-${extra}.txt"
done
```

`--python-version 3.10` is the **floor**, not a target: `--universal`
mode emits markers that select per-version dependencies up to the
highest interpreter the resolver can reach. Bumping the floor
requires a corresponding bump in `[project].requires-python`.

### Vulnerability scanning

The `audit` job in `.github/workflows/pr_gate.yml` runs
`pip-audit --strict --requirement ci/constraints-<extra>.txt` against
each constraint file on every PR. `--strict` fails the job if any
advisory can't be resolved (yanked release, network failure) so a
silent skip is not possible.

`pip-audit` itself is pinned (and hash-checked) via
`ci/constraints-audit.txt`. The job audits this file too, so the
auditor self-checks.

## GitHub Actions pinning

Every `uses:` line in `.github/workflows/` is pinned to a full 40-char
commit SHA with a trailing `# vX.Y.Z` comment giving the human-readable
release the SHA points at. Floating major tags (`@v6`) are forbidden.

Dependabot rewrites both the SHA and the comment when a new release is
published; PRs are grouped weekly to limit churn. Security advisories
bypass the schedule.

Manual regeneration (when not waiting for dependabot):

```bash
# Look up the release tag's SHA on the upstream repo.
git ls-remote https://github.com/<owner>/<repo>.git refs/tags/<vX.Y.Z>
```

then update the `uses:` line and the trailing comment.

The bootstrap script `.copilot/scripts/pin_actions.py` records every
pin the repo started with and is safe to re-run (it skips lines that
already match a known SHA).

## Downstream consumer template

[templates/c_abi_consumer/pyproject.toml](templates/c_abi_consumer/pyproject.toml)
pins `bocpy~=MAJOR.MINOR` as both a build requirement and a runtime
dependency. The compatible-release bound expresses the public C ABI
the template was authored against; the `finalize-pr` skill bumps it
in lock-step with the root version.

## Wheel SBOMs (PEP 770)

Every wheel produced by `.github/workflows/build_wheels.yml` ships a
[CycloneDX 1.6](https://cyclonedx.org/specification/overview/) JSON
SBOM embedded under `*.dist-info/sboms/bocpy.cdx.json`, in line with
[PEP 770](https://peps.python.org/pep-0770/).  Consumers can extract
it with any zip tool — see [`sphinx/source/sbom.rst`](sphinx/source/sbom.rst)
for end-user docs, including extraction recipes and validation
walk-throughs.

### How it gets there

Generation runs inside cibuildwheel's `CIBW_REPAIR_WHEEL_COMMAND_*`
hook on every platform leg:

| Platform        | Repair command                                                                                  |
|-----------------|--------------------------------------------------------------------------------------------------|
| `manylinux`/`musllinux` | `auditwheel repair -w {dest_dir} {wheel} && python {project}/scripts/build_sbom.py inject {dest_dir}` |
| `macos` (arm64 + intel) | `delocate-wheel --require-archs {delocate_archs} -w {dest_dir} -v {wheel} && python {project}/scripts/build_sbom.py inject {dest_dir}` |
| `windows`       | `python {project}/scripts/build_sbom.py inject {wheel} --copy-to {dest_dir}`                     |

Each leg also sets `CIBW_BEFORE_BUILD: "pip install tomli"` so the
generator can read `pyproject.toml` on cp310 (where stdlib `tomllib`
is not yet available).

The generator at [`scripts/build_sbom.py`](scripts/build_sbom.py) is
stdlib-only and:

1. constructs the CycloneDX 1.6 document deterministically from
   `pyproject.toml` plus `$GITHUB_SHA`,
2. opens the wheel, copies every entry except an existing
   `*.dist-info/sboms/bocpy.cdx.json` into a sibling temp file,
   appends the new SBOM, regenerates `RECORD`, and atomically renames
   the temp file over the original.

### How it gets verified

After the `merge` job assembles every wheel, the `verify_sboms` job
runs two checks:

1. **Structural validation** — `python scripts/validate_sbom.py sboms/`
   asserts the wire-format invariants bocpy commits to
   (`bomFormat == "CycloneDX"`, `specVersion == "1.6"`,
   `urn:uuid:<v4>` serial, ISO 8601 `Z` timestamp, the root
   component's `bom-ref` and `purl` both equal `pkg:pypi/bocpy@<v>`,
   the dependencies list references the root, etc.).  Stdlib-only on
   purpose so the validator has no third-party supply-chain surface
   of its own.
2. **Third-party scan** — `grype` is downloaded via
   `anchore/scan-action/download-grype` (SHA-pinned) and run against
   every extracted SBOM with `--fail-on high`.  Because bocpy ships
   no third-party runtime Python dependencies, the expected finding
   count is zero; a HIGH or CRITICAL match would fail the workflow.

The `merge` job also uploads a separate `sboms` artifact containing
one `<wheel-stem>.cdx.json` per built wheel, for downstream tools
that prefer flat files over re-extracting from each wheel.

## Dependabot

[.github/dependabot.yml](.github/dependabot.yml) covers three
ecosystems:

1. `pip` rooted at `/ci` — bumps inside the hashed constraint files.
2. `github-actions` rooted at `/` — bumps SHA-pinned `uses:` lines.
3. `pip` rooted at `/templates/c_abi_consumer` — bumps the template's
   `bocpy` pin when a new compatible release is published.

All three are scheduled weekly and grouped to one PR per ecosystem.
